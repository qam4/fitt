# Requirements: FITT Phase 3 — Telegram + Browser Interface

## Overview

Phase 3 makes FITT reachable from the user's phone (Telegram bot) and
from any browser (Open WebUI, self-hosted). Both are thin adapters
that forward messages to the Phase 1 gateway; neither duplicates
routing, auth, memory, or session logic. The gateway stays the
source of truth.

Telegram handles text, images (forwarded as multimodal content to
the gateway), and voice notes. Voice transcription and TTS reply are
deferred to Phase 8; in Phase 3 voice notes get an explicit "voice
not yet supported" reply rather than silent failure.

Open WebUI is added as a Docker compose service pointed at the
gateway's OpenAI-compatible endpoint. It provides a familiar
chat-bubble UI for longer sessions and multi-model comparison.

## User Stories

### 1. Telegram bot: basic chat

As a user, I want to message my FITT bot on Telegram and get a
reply generated by the gateway so I can chat from my phone.

#### Acceptance Criteria
- 1.1 The Telegram bot runs as a separate process on the Hub
  (systemd-style service, like the gateway).
- 1.2 Incoming Telegram messages from allowlisted user ids are
  forwarded as `POST /v1/chat/completions` to the gateway.
- 1.3 Messages from non-allowlisted users are silently ignored
  (logged but no reply).
- 1.4 The bot uses the `main` session by default.
- 1.5 The bot streams the gateway's reply back to Telegram by
  editing a sent placeholder message as tokens arrive.
- 1.6 The alias used for each request is configurable per-chat
  (stored in a small sqlite db or on-disk file keyed by chat id);
  default is `fitt-default`.

### 2. Telegram bot: allowlist

As the owner, I want only my Telegram user id to be able to talk
to the bot so strangers who stumble on the bot can't use my
resources.

#### Acceptance Criteria
- 2.1 The allowlist is a list of numeric user ids stored in
  `~/.fitt/secrets.yaml` under `telegram.allowlist_user_ids`.
- 2.2 An empty allowlist means "allow no one"; the bot logs and
  drops every message. Never "allow everyone" by default.
- 2.3 Any non-allowlisted update (message, voice, photo, command)
  returns no response and logs one line.

### 3. Telegram bot: session selection

As a user, I want to switch which session the bot is using so I
can segregate retro-ai chat from home-automation chat on my phone.

#### Acceptance Criteria
- 3.1 `/session` lists active sessions and marks the current one.
- 3.2 `/session <id>` switches the current chat's active session
  to `<id>`. If `<id>` isn't valid the bot replies with the list
  of active ids.
- 3.3 `/session new <id> [name]` creates a new session via the
  gateway's CLI-equivalent operation (direct registry call; the
  bot process has file access to the sessions dir).
- 3.4 The per-chat session-id preference persists across bot
  restarts.

### 4. Telegram bot: model selection

As a user, I want to switch which alias the bot uses per chat so
I can say "use fitt-smart for this conversation".

#### Acceptance Criteria
- 4.1 `/model` lists available aliases and marks the current one.
- 4.2 `/model <alias>` sets the current chat's alias. Invalid
  alias -> list of valid ones.
- 4.3 Preference persists across bot restarts.

### 5. Telegram bot: images

As a user, I want to send a photo to the bot and have it describe
it so I can ask things like "what is this error?" with a
screenshot.

#### Acceptance Criteria
- 5.1 An incoming photo with an optional caption is forwarded to
  the gateway as a multimodal message (OpenAI-format content parts
  with image_url data: URI + optional text).
- 5.2 If the caption is absent, a reasonable default prompt is
  used (e.g. "Describe this image.").
- 5.3 The gateway's response is returned as a normal text reply.

### 6. Telegram bot: voice notes (stub in Phase 3)

As a user, I want an explicit "not yet" when I send a voice note
so I don't wonder if the bot is broken.

#### Acceptance Criteria
- 6.1 Voice messages get a text reply: "Voice is not wired up
  yet. Phase 8 adds STT. Try a text message for now."
- 6.2 The voice note file itself is NOT downloaded or forwarded.

### 7. Open WebUI

As a user, I want a browser-based chat UI pointed at the gateway
so I can have longer, wide-screen conversations.

#### Acceptance Criteria
- 7.1 Open WebUI runs as a Docker container on the Hub on port
  3000, reachable via Tailscale.
- 7.2 It's pre-configured to use the gateway's OpenAI-compatible
  endpoint as its "OpenAI" provider.
- 7.3 A single admin account is created on first run. Signup is
  disabled thereafter.
- 7.4 Open WebUI's health is reachable at
  `http://<hub-tailscale-ip>:3000/` from any Tailscale device.
- 7.5 The firewall rule (inbound TCP 3000, Private profile only)
  is created / updated by the install script.

### 8. Persistent bot preferences

As a developer, I want the bot's per-chat preferences stored in
one place so restarts don't reset everything.

#### Acceptance Criteria
- 8.1 Per-chat preferences (alias, session, any future setting)
  live at `~/.fitt/telegram/prefs.json`.
- 8.2 The file is atomically written (tempfile + os.replace)
  after every change.
- 8.3 Corrupted JSON is logged + treated as "no preferences set"
  rather than crashing the bot.

## Non-Goals (explicit)

- **No voice transcription** (Phase 8).
- **No TTS reply** (Phase 8).
- **No Telegram inline-keyboard tool approval** (Phase 4 decides
  that with MCP in hand).
- **No multi-user Telegram** (single owner, allowlist is a safety
  net, not a multi-tenancy feature).
- **No per-chat identity overrides** (identity stays global).
- **No Open WebUI integration with FITT memory-specific features**
  (Open WebUI has its own memory concepts we're not adopting).

## Shareable-by-construction

- Bot token and allowlist ids live in `secrets.yaml`, never in git.
- Per-chat prefs live in `~/.fitt/telegram/`, outside the repo.
- Open WebUI's admin password is set at first-run by the user, not
  shipped as a default.
- The bot's defaults are safe: empty allowlist = no access.
